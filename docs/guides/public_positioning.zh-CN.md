# ClawGraph 对外版产品故事（2026-04）

这份文档只回答四个问题：

1. ClawGraph 是什么
2. 它解决什么问题
3. 当前已经做到哪一步
4. 当前不应该怎么宣传

## 一句话定位

ClawGraph 是一层把真实 agent 运行自动沉淀为训练数据、验证资产和替代建议的控制面。

## 它不是什么

- 不是 agent runtime
- 不是训练引擎
- 不是通用 tracing 平台
- 不是已经完成发布的托管训练平台

## 它为谁而做

- 平台与 Runtime 团队：希望以最小侵入方式接入真实运行
- 数据与训练团队：希望把运行整理成可训练数据和评测资产
- 评估与质量团队：希望比较候选模型、基线模型和替代建议
- 交付与解决方案团队：希望向客户演示一条清晰、可追溯的学习闭环

## 它解决的核心问题

很多 agent 系统已经能跑，但训练与替代阶段常见几个断层：

- 线上只有平日志，没有可训练、可复核的结构化数据
- retry、fallback、subagent 这类关键结构容易在日志里丢失
- 调试、训练、评测各用一套数据，无法回到同一条真实运行
- 人工判断、judge 结果和原始执行混在一起，难以审计
- 训练系统需要稳定的输入文件，上游 runtime 只给临时日志

ClawGraph 的思路是把这些断层拆开：

1. 先 capture 真实运行
2. 再整理出 run / branch / replay 视图
3. 再追加可治理的 supervision 和 review 结果
4. 最后导出训练数据、验证资产和替代建议

## 当前可以稳定讲的能力

- 通过 proxy 接入真实 agent 或 benchmark 流量
- 把运行沉淀为可回放、可复核、可治理的执行图数据
- 自动形成数据准备、人工复核、数据筛选和数据集导出闭环
- 用统一对象链路查看 `dataset -> candidate -> eval -> promotion`
- 为候选模型生成替代建议和 rollback 条件

## 当前需要明确写清的边界

- 训练执行由外部系统负责，例如 `Logits`
- 当前 Web 已经能查看训练资产血缘，但还不是完整的训练执行工作台
- 首方 HTTP API 已可联调，但权限、安全和性能验证仍需继续补齐
- 当前稳定评测路径仍以离线验证为主，通用 env / verifier-driven RL 评测仍是后续建设项

## 建议的对外说法

一句话：

- 把真实 agent 运行变成训练数据、验证资产和替代建议。

三句话展开：

- 先通过 proxy 接入真实运行，不要求先改业务逻辑。
- 再把运行自动整理成可训练数据、人工复核队列和验证套件。
- 最后在统一控制面里比较候选模型与基线模型，并给出替代建议。

## 不建议的说法

- “完整托管训练平台”
- “训练、评测、放量全部产品化闭环”
- “已经覆盖完整的在线 RL 与多租户生产治理”

这些说法会超出当前实现边界。

## 当前适合的发布口径

- 高级技术预览
- 定向合作预览
- self-hosted 工程控制面 demo

## 对外演示时建议的 5 分钟路径

1. 展示真实流量如何通过 proxy 进入系统
2. 展示最近运行、回放和步骤摘要
3. 展示数据准备、人工复核和数据筛选如何形成数据集
4. 展示训练资产、离线验证和替代建议链路
5. 最后明确当前边界：训练执行由外部系统负责，发布级安全和性能仍在继续补齐
